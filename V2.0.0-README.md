# cc370 — GCC 3.4.6 for i370-ibm-mvspdp (provenance)

cc370 is based on **GCC 3.4.6** for the `i370-ibm-mvspdp` target (`TARGET_PDPMAC`:
HLASM/EBCDIC, `COPY PDPTOP` / `PDPPRLG` / `PDPEPIL`, libc370-compatible calling
conventions). This file records where the GCC fork came from.

The earlier GCC 3.2.3-based generation lives in the frozen `mvslovers/c2asm370`
repo (**v1.x**), kept as a fallback for projects still on the old name.

## Provenance

Snapshot of `mvslovers/i370-gcc` @ `feature/ebcdic-char-constants` (commit
`0710af0c`), itself a fork of `linas/i370-gcc` (GCC 3.4.6). Brought in here as
v2.0 because the upstream PRs to linas went unanswered, so we develop the
compiler against our own requirements from now on.

Key mvspdp work carried in this snapshot:

- **EBCDIC charset (CP037 + ecosystem NEL):** character constants and string
  literals map to CP037; the newline `\n` maps to EBCDIC NEL `0x15` (byte-
  identical to the ecosystem httpd `cp037_atoe`/`cp037_etoa`, so it matches
  mvsMF source upload and HTTP output); `\x..`/`\NNN` escapes stay binary-literal.
- **Codegen for IFOX00:** fixed-column assembler text, valid MVS symbol names
  for private statics, signed 32-bit literals.
- **Host portability:** builds on macOS and Linux, x86-64 and ARM64, GCC 14 hosts.

## v2.0 status

- ~~Strip all non-i370/mvspdp targets from the GCC tree~~ **done** — slimmed from
  20640 to ~1095 tracked files (removed other GCC targets, other-language
  frontends, the test suite, and other-language runtime libs); builds cc1 + xgcc.
- ~~asm()-extern **address-of** bug~~ **fixed** — `&x[i]` of an `asm()`-named
  extern no longer emits invalid `=V(*NAME)` (the leading `*` is stripped in the
  i370 backend address-of paths).
- **Optimization: `-O1` only.** `-O2`/`-Os`/`-O3` are UNSAFE on this backend —
  at `-O2`+ the `-funit-at-a-time` DCE drops `static` tables held by a global
  pointer (→ dangling `=V`, IFOX RC=8), and `-Os` additionally miscompiles the
  rexx parser (loops → S322). `-O1` is validated correct (rexx370 TSTALLB 84/84).
  This — not the old "3.2.3 memory" note — is the real reason for "-O1 only".

**Beyond the compiler, v2.0 grew a full host-native MVS cross-toolchain** so the
only thing that touches MVS is the final load module: **as370** (`.s` → OS/360
OBJ, byte-identical to IFOX00), **ld370** (OBJ → load module, replaces IEWL) +
**ar370** (object-deck `.a` archives for automatic library call) + the
host→MVS **`--unload`/`--xmit`** transport. Proven end-to-end on real MVS. See
`CLAUDE.md` (tool table + roadmap A–E), `as/README.md`, `docs/*-format.md`, and
`docs/roadmap-integration.md` (plain-language status + integration plan).

The host-portability work (amd64/arm64/gcc14) is carried in this tree.

## Building (cross-compiler, C only, on a modern macOS/Linux host)

GCC 3.4.6 is old K&R-ish C; modern clang/gcc must be told not to error on it:

```sh
CF="-g -O0 -fcommon -std=gnu89 -Wno-implicit-int -Wno-implicit-function-declaration \
    -Wno-int-conversion -Wno-error -Wno-return-type -Wno-deprecated-non-prototype"
mkdir build && cd build
CFLAGS="$CF" CFLAGS_FOR_BUILD="$CF" ../configure \
  --target=i370-ibm-mvspdp --enable-languages=c \
  --disable-threads --disable-nls --disable-shared --without-headers \
  --with-gcc-version-trigger=../gcc/version.c
make all-gcc CFLAGS="$CF" CFLAGS_FOR_BUILD="$CF"
```

Produces `build/gcc/cc1` (the compiler) and `build/gcc/xgcc` (the driver).
